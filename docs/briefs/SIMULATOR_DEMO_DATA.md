# Simulator demo data — a simulated athlete, not a fake screen

**Problem.** The iOS Simulator has no Apple Watch and no Oura, so HealthKit is
empty. With no observations the home correctly shows the locked F1 honest-silence
line ("We need more data to predict recovery."). That is *right* behaviour, but
it makes the rest of the app impossible to build, demo, or screenshot.

**The rule we will not break.** We never hardcode a readiness score, a fatigue
state, or Josi prose on the display side. That would be fake reality — and it
would violate architecture rule 3 (Dart is display only) *and* the locked
honest-silence rule. The bright line:

| | Honest | Fake reality |
|---|---|---|
| Synthetic **INPUT** → real engine | ✅ simulated athlete | |
| Hardcoded **OUTPUT** in Dart | | ❌ faked screen |

## How it works

We feed a coherent, synthetic season of **raw biometrics** through the real
engine **normalizer + Viterbi pipeline**, and let the real engine decide
everything:

```
demo_season.json  →  normalizeObservation(vendor:'apple')  →  processObservation  →  saveState/writeViterbiState
 (synthetic INPUT)        (real FFI normalizer)                  (real Viterbi advance)    (real state persistence)
```

The only synthetic thing is the biometric stream — exactly as it would be on a
bench test. This is the on-device twin of the rust `dev_sim` and python
`realworld_sim` seasons: **synthetic INPUT, real ENGINE.**

> **Honest scope (WU3, 2026-07):** the seeder drives the *engine* pipeline
> (normalizer → Viterbi → state persistence), but it is **not byte-identical to
> the production health-ingest path** — it does **not** call `writeBiometric` or
> write raw-observation rows to the vault the way `lib/services/health_ingest.dart`
> does (vault-first). So it warms the Viterbi state faithfully, but the
> raw-observation / biometric vault tables it leaves emptier than a real device
> would. Do not use it to validate the vault-write leg; use it to exercise
> readiness/advisor surfaces.

- **Fixture:** `assets/debug/demo_season.json` — a 30-day arc
  (base → overload → illness dip → recovery), mirroring `dev_sim`. Generated by
  `scripts/gen_demo_season.py` (re-run it if the arc changes). It stores raw
  HealthKit-shaped scalars + a day **offset** (not absolute dates), because the
  Viterbi monitor uses calendar-day windowing (42-day window, stale data
  excluded) — the seeder assigns real dates **ending today** at replay time so
  the season is never stale.
- **Seeder:** `lib/debug/demo_seeder.dart` — replays the season through the real
  FFI calls. It authors no readiness/state of its own; it only persists the
  engine's own `saveState()` output.

## How to use it

1. Build + run on the simulator — see **`README.md` → Quick start** for the
   xcframework + `flutter run` recipe.
2. In the app: **You → Settings**. In a **debug build only**, a
   *"Developer · Demo data"* section appears at the bottom.
   - **Seed ~10 days** → mid-calibration ("still learning you").
   - **Seed full season (~30 days)** → warmed-up: full readiness, the 4-axis
     "why", advisor variety.
3. Pull-to-refresh Today (or restart) to see the engine's result.
4. **Reset to honest cold-start:** Settings → *Delete All My Data* (the existing
   crypto-erase) → re-onboard. Back to F1 silence.

## The no-destroy guarantee

- Every call site is gated behind `kDebugMode`, so the seeder, the Settings
  section, and the asset wiring are **compiled out of release builds entirely**.
- Seeding only ever **adds** real-pipeline observations to the vault — the same
  bytes a watch would write. It cannot corrupt the production flow, and once a
  real device starts syncing, its data flows through the identical path.
- A real release build starts empty and stays honest until a real device feeds
  it.
