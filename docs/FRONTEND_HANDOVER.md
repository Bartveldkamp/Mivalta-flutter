## 1. Orientation

MiValta is a privacy-first, **100% on-device** AI fitness-coaching app. This repo
is the production Flutter frontend; it replaces the older `mivalta-android-client`
Kotlin app.

**The one contract that governs everything:**

> **The Rust engine DECIDES. Flutter DISPLAYS.**

Every number, label, zone, and prose string on screen is produced by the Rust
engine and rendered verbatim. There is **no coaching math, no thresholds, no
fallback logic in Dart**. When the engine has no answer, the UI shows the
engine's own "honest empty" signal (e.g. the locked F1 no-data copy), never a
fabricated value.

**Current milestone — MVP-1** (`docs/MVP1_BUILD_BRIEF.md`). Scope: **Monitor +
Advisory (workout suggestions only) + multi-vendor data ingest**, dark-first,
three-zone PULL home. The conversational AI / on-device LLM ("model W") is
**deferred to a later phase** and is not present in the production UI on this
branch. There is no `ReplanEngine` (plan editing) and no plan/periodization UI in
MVP-1.

### The 8 architecture rules (from `CLAUDE.md`, restated)

1. **No engine logic in Dart.** Computation stays in Rust.
2. **FFI is pure transport.** `rust/src/api.rs` binds one `gatc_ffi::*` call per
   function and returns the raw JSON string. No engine logic in the shim.
3. **Dart is display only.** Widgets map engine output to UI state. No
   thresholds, math, or fallback.
4. **SourceTier color tokens are LOCKED:** Medical `#2BD974`, Device `#00C6A7`,
   Partial `#E6872F`, Manual `#878C8C`. Use the token; never hardcode hex.
5. **F1 no-data copy is LOCKED verbatim:** "We need more data to predict
   recovery." Do not paraphrase or soften.
6. **No cloud round-trips in the LLM path.** On-device only. The only network I/O
   is pulling the user's own wearable data and (deferred phase) a first-launch
   model download.
7. **No dead code.** Every new public Dart symbol has a production call site
   within one PR.
8. **New behaviour needs a test** with a concrete-value assertion (`flutter test`
   / widget / integration).

## 3. How Flutter consumes the engine

The full path for any engine call:

```
Widget (lib/screens/*)
  → RustEngineBinding (lib/rust_engine.dart)          ← idiomatic Dart facade
    → generated FRB bindings (lib/src/rust/api.dart)  ← AUTO-GENERATED, do not edit
      → rust shim (rust/src/api.rs)                   ← one gatc_ffi::* call per fn
        → gatc-ffi (mivalta-rust-engine, git-pinned)  ← the engine, returns JSON
```

- The shim functions in `rust/src/api.rs` each delegate to exactly one
  `gatc_ffi::*` method and return the raw JSON string. No engine logic is added
  in the shim (the only shim-side work is bundling engines into the opaque
  `EnginesHandle`, parsing `athlete_id`, and — for `recommend_workout*` — reading
  the engine's *own* readiness signals back into the engine; still pure
  transport, no coaching decisions).
- **FRB codegen.** `flutter_rust_bridge_codegen` reads `flutter_rust_bridge.yaml`
  (`rust_input: crate::api`, `rust_root: rust/`, `dart_output: lib/src/rust`) and
  regenerates everything under `lib/src/rust/`. You never edit those files by hand.
- **The `.so`.** The shim crate compiles to `libmivalta_rust_bridge.so` (cdylib),
  packaged into the APK under `android/app/src/main/jniLibs/`. `gatc-ffi` is
  statically linked into it (rlib) — the stray `libgatc_ffi-*.so` left in the
  jniLibs dir is deleted in the build (see `README.md`). FRB init is one-shot and
  Android-only (`RustEngineBinding.ensureRustInit()` throws on non-Android hosts).
- **The engine pin.** `rust/Cargo.toml` pins `gatc-ffi` (and `gatc-viterbi`, for
  the `UniversalObservation` type used by `process_manual_observation`) to a
  specific `mivalta-rust-engine` git revision over SSH (private repo; CI uses the
  `RUST_ENGINE_DEPLOY_KEY` deploy key). On this branch the pin is **`b603b5e`**,
  and the rust-engine `engine_registry.json` at that point is **v2.24**. (See §10
  — `CLAUDE.md`/`README.md` quote older pins.) Local-dev can override the SSH pin
  with a path patch in `.cargo/config.toml` (see `README.md`).

### Adding a new engine method end-to-end

This crosses the Dart↔Rust boundary, which is **load-bearing for the whole
product**. Per `CLAUDE.md` scope discipline you **must pause and surface the
proposed change** before editing `rust/src/api.rs` or `lib/rust_engine.dart`.

1. **Shim** — add a `pub fn` in `rust/src/api.rs` that calls one `gatc_ffi::*`
   method and returns its JSON `String` (or scalar), mapping errors via
   `.map_err(Into::into)`.
2. **Regen** — run `flutter_rust_bridge_codegen` on the build executor to
   regenerate `lib/src/rust/`. (Until regen runs, a new shim fn has no Dart
   caller and the crate-local dead-code lint can't see it.)
3. **Facade** — add a thin method on `RustEngineBinding` in `lib/rust_engine.dart`
   that calls the generated `rust_api.*` function.
4. **Widget** — call the facade method from a screen, `jsonDecode`, and render.
   Add a contract test in `test/` and/or a widget test.
5. If the new method changed the engine pin, bump `rust/Cargo.toml`, regenerate
   `Cargo.lock`, and rebuild the `.so`/APK on the executor.

## 4. What the engine provides today

The shim bundles **7 of the 16 rust-engine FFI engines** into `EnginesHandle`:
`ViterbiEngine`, `AdvisorEngine`, `VaultEngine`, `DashboardEngine`,
`NormalizerEngine`, `CpEngine`, `PostProcessEngine`. (The rust-engine also exposes
TablesEngine, PlanEngine, ChatEngine, GuardrailsEngine, ReplanEngine, MmpEngine,
WbalEngine, DecouplingEngine — none are bound in this frontend on this branch.)

Key payloads the frontend consumes:

- **`readiness_indicator`** — the headline 4-axis readiness blend (HMM posteriors
  + Banister + physio + psychological): `{score, level, confidence, …}`.
- **Dashboard widgets** — `get_state_widget` / `get_session_widget` /
  `get_context_widget` drive the three home zones (state recommendation,
  today's session, ACWR/monotony/strain + alerts). `get_dashboard` is the
  composite version.
- **Advisor A/B/C** — `recommend_workout` / `recommend_workout_with_history`
  return three equal-weight options (no ranking; the athlete chooses).
- **Analytics** — `read_readiness_history`, `read_daily_loads`,
  `read_biometric_history`, `read_mmp_history`, `fit_cp`, `compute_time_in_zone`,
  `recent_decoupling_pct`, `fitness_series`, `get_workout_detail`,
  `completed_workout_facts` + `build_post_workout_report`.
- **Source tiers** — `last_observation_source_tier` returns one of
  `Medical`/`Device`/`Partial`/`Manual` (or `null`).

> **Source of truth for method signatures and JSON schemas:** the rust-engine
> repo's **`docs/frontend/FFI_API_CONTRACT.md`**
> (<https://github.com/Bartveldkamp/mivalta-rust-engine>). Do not duplicate
> schemas here — go there for the authoritative field-by-field contract. This doc
> only maps which contract methods are wired into the Flutter UI.

## 6. Continuity & state

ViterbiEngine state must survive app restarts (the learned HMM, ceiling
intelligence, OutcomeTracker, etc.). The pattern, all driven from
`ReadinessScreen._fetch()`:

1. On launch, `readPersistedState(athleteProfileJson, vaultPath)` checks the vault
   for a saved blob.
2. **State exists →** `constructEnginesFromState(…, viterbiStateJson)` restores it.
   If the blob is corrupt/incompatible, the screen falls back to
   `constructEnginesFresh`, overwrites the bad blob, and surfaces a one-time,
   non-silent "history reset" snackbar (never silent).
3. **No state (first run) →** `constructEnginesFresh(…)`, then immediately
   `saveState` → `writeViterbiState` to persist.
4. After any state-changing op, and on app background/detach
   (`didChangeAppLifecycleState`), `saveState` → `writeViterbiState` again.
   `paused` is the reliable save window; `detached` is best-effort.

The vault lives at a **persistent** path (`<appSupport>/mivalta-vault`). The
athlete profile itself lives in the encrypted vault; only a random `athlete_id`
UUID is held in the plaintext pointer.

## 7. Design system

- **Tokens:** `lib/theme/tokens.dart` — `MivaltaColors`, `MivaltaSpace`,
  `MivaltaRadii`, `MivaltaGlass`, `MivaltaMotion`, plus the dark-first
  `mivaltaDarkTheme()` and the engine-string→color mappers
  (`readinessLevelColor`, `fatigueStateColor`). Most color tokens are placeholders
  pending the design pass; **screens read `Theme.of(context)` / token constants
  and never hardcode color or type.**
- **LOCKED SourceTier tokens:** `lib/theme/source_tier.dart` — Medical `#2BD974`,
  Device `#00C6A7`, Partial `#E6872F`, Manual `#878C8C` (in `kSourceTierHex`).
  These mirror the rust-engine `DataSourceTier` variants and are change-controlled
  — a value changes only on a founder-authorised token bump in the same PR.
  `sourceTierFromEngine()` maps the engine's PascalCase JSON variant onto the enum,
  returning `null` for anything unknown so the caller falls through to the no-data
  path.
- **LOCKED F1 copy:** `lib/copy/f1.dart` — `kF1NoDataCopy = "We need more data to
  predict recovery."` Single source of truth; do not paraphrase or soften.

## 8. Build & test

```bash
flutter pub get                                            # Resolve deps
flutter analyze                                            # Static analysis (local)
flutter test                                               # Unit/widget tests
flutter build apk --debug --target-platform android-arm64  # Debug build (Android)
flutter run                                                # Launch on attached device
```

Building the `.so` (the engine bridge) is a separate Rust step via `cargo ndk`
against the pinned engine rev — see `README.md` for the exact `cargo ndk … build`
invocation, the NDK version, and the `libgatc_ffi-*.so` cleanup line.

**Critical CI lesson — local-green ≠ CI-green.** `.github/workflows/ci.yml` runs
`flutter analyze --fatal-infos`, which is **stricter** than a bare local
`flutter analyze` or `flutter test`: an *info*-level lint that passes locally will
**fail CI**. `smoke-build.yml` additionally resolves the engine pin, builds the
`.so`, and asserts the expected libraries + `compiled_tables.json` are packed into
the APK. Run `flutter analyze --fatal-infos` locally before pushing.
